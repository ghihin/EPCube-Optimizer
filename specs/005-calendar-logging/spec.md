# Feature Specification: Googleカレンダーへの自動実行結果ロギング

**Feature Branch**: `005-calendar-logging`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "FUTURE_ENHANCEMENTS.md で定義されていた「実行履歴の可視化」を解決するため、アプリ内データベースではなく「Googleカレンダーへの予定（イベント）書き込み」を採用する。"

---

## User Scenarios & Testing *(必須)*

### User Story 1 - 成功時の実行結果をカレンダーで確認する (Priority: P1)

ユーザーは翌朝スマートフォンのカレンダーアプリを開くと、前夜（21:00頃）に自動実行されたEPCubeの設定結果が予定として記録されている。タイトルの「✅ EPCUBE設定完了」とその説明欄を見ることで、「どのモードに設定されたか」「なぜそのモードになったのか（天気予報の根拠）」「変更前のSOCはいくつだったか」を一目で把握できる。

**Why this priority**: 本機能の核心的な価値であり、ユーザーが「なぜこのSOCに設定されたのか？」という疑問を解消する主要なシナリオ。

**Independent Test**: 深夜の自動実行をテストモードで手動実行し、成功すると指定のGoogleカレンダーに「✅ EPCUBE設定完了」の予定が作成されることで検証可能。

**Acceptance Scenarios**:

1. **Given** 自動実行が正常に完了した状態で、**When** ユーザーがGoogleカレンダーを開いたとき、**Then** 実行日時に「✅ EPCUBE設定完了」というタイトルの15分間の予定が存在する。
2. **Given** 上記の予定が作成されているとき、**When** ユーザーが予定の詳細を開いたとき、**Then** 説明欄に「設定モード」「目標SOC」「変更前のSOC」「運転モード」「翌日気象条件（天気・雲量・降水確率）」が記載されている。
3. **Given** グリーンモード（深夜充電なし）が設定された場合、**When** ユーザーが説明欄を確認したとき、**Then** 目標SOCは「(深夜充電なし)」と表示されている。
4. **Given** スマートモード（深夜充電あり）が設定された場合、**When** ユーザーが説明欄を確認したとき、**Then** 目標SOCは具体的な数値（例: 85%）として表示されている。

---

### User Story 2 - 失敗時のエラー内容をカレンダーで確認する (Priority: P2)

何らかの理由（ネットワーク障害、天気API取得失敗、マクロ操作失敗など）で自動実行が失敗した場合にも、ユーザーはカレンダーに記録された「❌ EPCUBE設定失敗」の予定を見て、エラーの発生を把握し、その原因を説明欄で確認できる。

**Why this priority**: 失敗を検知できなければ、翌日も誤った充電設定のままになるリスクがある。エラー通知と可視化は安心して運用するために不可欠。

**Independent Test**: エラーが発生するシナリオ（例: ネットワーク遮断）を意図的に再現し、「❌ EPCUBE設定失敗」の予定が記録されることで独立して検証可能。

**Acceptance Scenarios**:

1. **Given** 自動実行がエラーで終了した状態で、**When** ユーザーがGoogleカレンダーを開いたとき、**Then** 実行日時に「❌ EPCUBE設定失敗」というタイトルの15分間の予定が存在する。
2. **Given** 上記の失敗予定が作成されているとき、**When** ユーザーが詳細を開いたとき、**Then** 説明欄の「実行結果」欄にエラーの理由が記載されている。
3. **Given** カレンダーへの書き込み自体が失敗した場合、**When** 書き込みを再試行したとき、**Then** 二重でイベントが登録されない（べき等性が保たれる）。

---

### User Story 3 - 過去の実行履歴を時系列で振り返る (Priority: P3)

ユーザーはGoogleカレンダーの週表示・月表示を使い、過去数週間にわたるEPCubeの設定履歴を時系列で振り返ることができる。これにより「先週ずっとグリーンモードが続いた」「この日の失敗以降に充電量が変わった」といった傾向を把握できる。

**Why this priority**: 蓄積されたカレンダーイベントが自然と履歴になるため、本機能を実装することで自動的に得られる価値。ユーザーにアプリを開かせずに振り返りを提供できる。

**Independent Test**: 複数の実行結果がカレンダーに記録された後、カレンダーの週表示で複数の予定が時系列で表示されることで検証可能。

**Acceptance Scenarios**:

1. **Given** 過去7日間に自動実行が毎日実行された状態で、**When** ユーザーがGoogleカレンダーの週表示を開いたとき、**Then** それぞれの実行日時に「✅ EPCUBE設定完了」または「❌ EPCUBE設定失敗」の予定が時系列で表示される。

---

### Edge Cases

- カレンダーへの書き込み権限が付与されていない場合、ユーザーに権限付与画面を案内する（またはエラーログに記録する）。
- Androidの機内モードやDozeModeにより、カレンダーへの書き込みAPIがタイムアウトする場合は書き込みをスキップし、通常の自動実行処理には影響を与えない。
- 同一実行に対してカレンダーイベントが重複登録されないようにする（再試行ロジックを持つ場合も含む）。
- 記録するGoogleカレンダーが削除・無効化されていた場合、エラーをログに記録してサイレントに失敗する。
- SOC値や天気データが取得できなかった場合（部分的な情報しかない場合）は、取得できた情報だけで説明欄を記録する。

## Requirements *(必須)*

### Functional Requirements

- **FR-001**: 深夜の自動実行（天候取得〜マクロ操作）が完了またはエラーで終了した直後に、ユーザーのGoogleカレンダーへイベントを自動登録しなければならない。
- **FR-002**: 実行成功時のイベントタイトルは「✅ EPCUBE設定完了」、実行失敗時のイベントタイトルは「❌ EPCUBE設定失敗」でなければならない。
- **FR-003**: イベントの時刻は自動実行を開始した時刻（例: 21:00）とし、15分間の予定として登録しなければならない。
- **FR-004**: イベントの説明欄には次の情報を含まなければならない：
  - 設定結果（設定モード、目標SOC、または失敗時のエラー理由）
  - 変更前の状態（マクロ実行直前に画面から読み取ったSOC値、運転モード）
  - 気象条件（翌日の天気、雲量、降水確率）
- **FR-005**: 深夜充電なし（グリーンモード）の場合、目標SOCの欄には「(深夜充電なし)」と明示しなければならない。
- **FR-006**: カレンダーへの書き込みがサイレントに失敗した場合でも、メインの自動実行フロー（SOC設定・マクロ操作）は中断されてはならない。
- **FR-007**: イベント登録には、ユーザーが事前に許可を与えたGoogleアカウントのプライマリカレンダー（またはユーザーが指定したカレンダー）を使用しなければならない。

### Key Entities

- **AutomationResult（自動実行結果）**: 1回の自動実行の成否・設定モード・目標SOC・エラー理由を表す。
- **PreExecutionSnapshot（実行前スナップショット）**: マクロ実行直前に読み取った現在のSOC値と運転モードを表す。
- **WeatherContext（気象コンテキスト）**: 判定の根拠となった翌日の天気・雲量・降水確率を表す。
- **CalendarEvent（カレンダーイベント）**: 上記3つの情報をもとに生成され、Googleカレンダーに書き込まれる予定エントリ。

## Success Criteria *(必須)*

### Measurable Outcomes

- **SC-001**: 自動実行が完了してから5秒以内にGoogleカレンダーへの予定登録が完了する。
- **SC-002**: 連続7日間の自動実行において、カレンダーイベントの登録成功率が95%以上である（カレンダー側の問題を除く）。
- **SC-003**: カレンダー登録が失敗した場合でも、EPCubeのSOC設定成功率に影響を与えない（独立して失敗する）。
- **SC-004**: ユーザーが翌朝カレンダーを開いたとき、前夜の実行結果と判定根拠を1画面内で確認できる（説明欄が400文字以内に収まっている）。
- **SC-005**: 1回の自動実行に対してカレンダーイベントが必ず1件だけ登録される（重複登録なし）。

---

## Assumptions

- ユーザーはすでにGoogleアカウントをAndroidデバイスに設定・ログインしており、アプリはそのアカウントへのカレンダー書き込み権限を取得済みである（または初回起動時に権限要求を実施済みである）。
- 自動実行は毎晩21:00頃にAlarmManager経由で起動され、カレンダーへの書き込みはその後の処理として実行される。
- カレンダーへの書き込みにはAndroid標準の `CalendarContract` (ContentResolver) を使用する（外部カレンダーSDKは使用しない）。
- 記録先のカレンダーはプライマリアカウントのデフォルトカレンダーとする（将来的にユーザーが選択できるよう拡張する余地を残す）。
- `PreExecutionSnapshot`（SOC値・運転モード）の取得はAccessibilityServiceマクロがすでに担っており、本機能はその結果を受け取って利用する。
- エラーが発生した場合のカレンダー書き込み再試行は最大1回とし、それでも失敗した場合はサイレントに終了する。
- アプリはバックグラウンド（WakeLock）での処理が許可されている（003-automation-scheduling で対応済み）。
