# 仕様書: Weather Logic Accuracy Upgrade (Open-Meteo Integration)

## 1. 目的
現在の「雲量（Cloudiness）」に依存する充電量判定ロジックからの脱却。「夜に曇りで昼が快晴」のような日照のゴールデンタイムを正確に評価するため、気象データ取得元を `Open-Meteo API` へ移行し、太陽光パネル発電量に直結する**「短波放射量（Shortwave Radiation: W/m²）」**を判定ロジックのコアに据えて予測精度を劇的に向上させる。
**【対象シーズン】**: 本仕様は、エアコン利用がほぼ不要な「春〜初夏（および秋の中間期）」の運用を主な前提とした最適化である。

## 2. API移行要件
* **移行先**: Open-Meteo API (`https://api.open-meteo.com/v1/forecast`)
* **必須リクエストパラメータ**:
  * `latitude=35.93` (鶴ヶ島市周辺の緯度を設定・定数化)
  * `longitude=139.40` (鶴ヶ島市周辺の経度を設定・定数化)
  * `hourly=shortwave_radiation` (短波放射量を1時間ごとに取得)
  * `timezone=Asia/Tokyo`
* **廃止対象**: 既存の `OpenWeatherMap API` の呼び出し、および関連する設定値（APIキー等）のクリーンアップ。

## 3. 判定ロジック（日射量の評価）
* `CalculateTargetSocUseCase` を改修し、「翌日の日射量」と「時期（季節）」「スケジュール（通勤/在宅）」を掛け合わせたハイブリッド判定に変更する。

### 3.1 季節ごとの SOC 決定マトリクス
対象日の「月（Month）」によってエアコン利用や日照条件が変わるため、以下の3シーズンに分けてマトリクスを切り替える。
いずれの季節も気象判定には Open-Meteo の積算日射量を評価する。現在SOC（夜間のバッテリー残量）はすべての季節において評価から除外（無視）する。

#### A. 中間期（春〜初夏・秋: 4〜6月, 10〜11月 等）
エアコン利用がほぼ不要なため、アグレッシブに深夜充電を抑えるマトリクス。積算日射量を3段階で判定。
* **晴れ (`THRESHOLD_SUNNY` 以上)**: 強制 60%
* **くもり (`THRESHOLD_CLOUDY` 以上 〜 未満)**: 在宅 70% / 出社 60%
* **悪天候・雨 (`THRESHOLD_CLOUDY` 未満)**: 在宅 100% / 出社 80%

#### B. 夏季（7〜9月）
冷房を夜間も利用するため、深夜の安い電力で多めに充電しておくマトリクス。積算日射量を2段階（晴れ or 非晴れ）等で判定。
* **晴れ (`THRESHOLD_SUNNY` 以上)**: 在宅 80% / 出社 60%
* **非・晴れ (`THRESHOLD_SUNNY` 未満)**: 在宅 100% / 出社 90%

#### C. 冬季（12〜3月）
暖房利用が多く日照量も少ないため、最も安全余裕（多め）を持たせたマトリクス。
* **晴れ (`THRESHOLD_SUNNY` 以上)**: 在宅 100% / 出社 70%
* **非・晴れ (`THRESHOLD_SUNNY` 未満)**: 在宅 100% / 出社 90%

### 3.2 判定方法の詳細
  1. **日照時間の考慮**: 翌日（対象日）の日照時間帯（例: 08:00 〜 16:00）における1時間ごとの `shortwave_radiation` の値を合計し、「積算日射量（W/m²・h相当）」として算出する。
  2. **定数化要件**: 各閾値、および目標SOCの数値は季節ごとの定数として管理し、チューニングを容易にする。

### 3.3 エラー時のフォールバック（フェイルセーフ）
気象API（Open-Meteo）の取得に失敗した場合や例外が発生した場合は、最も安全な【全シーズン共通で目標SOC 100%（スマートモード）】として動作させるフェイルセーフを設ける。

## 4. 既存機能との互換性・影響範囲
* **ログ出力の更新**: 取得した気象データについて、`AutomationExecutionLog` や `CalendarLogWriter`（Googleカレンダーへの書き込み）の「説明欄」に記載している雲量情報を廃止し、「予想日射量（W/m²等）」といった新しい指標で記載する。
* **安全性確保**: 既存の AccessibilityService による「マクロ自動実行」や「カレンダー書き込みのインフラ」を破壊しないよう、ドメインモデル（`WeatherForecast` など）の切り替えや移行を慎重に行うこと。
