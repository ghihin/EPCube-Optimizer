## User Scenarios & Testing

### User Story 1 - 深夜の全自動SOC最適化と設定反映 (Priority: P1)

ユーザーは、毎晩指定時刻（例：23:30）にシステムが自動的に翌日の気象予報とスケジュールを取得し、最適なSOCを計算して蓄電池システム（EPCube）へ設定することを期待する。これにより、毎日の手動操作の手間が完全に排除される。

**Why this priority**: 本プロジェクトの最大の目的である「手動操作の完全排除」を実現するコア機能であり、最も価値が高い。

**Independent Test**: 特定の時刻（数分後など）にスケジュール設定し、端末がスリープ状態であっても自動的に天気とカレンダーを取得、計算結果がマクロを通じて設定されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 蓄電池システムが接続されており、現在時刻が23:30になった時、 **When** 自動処理が開始される、 **Then** 翌日の気象予報と予定に基づく最適SOCが計算され、マクロによって蓄電池に正しく設定（20秒待機後の値再確認で成功判定）される。
2. **Given** 端末が画面オフ・スリープ状態である時、 **When** 23:30の自動処理トリガーが発火する、 **Then** 端末が一時的にスリープ解除（画面点灯・ロック解除等の適切な制御）され、マクロがフォアグラウンドで正常に実行完了する。

### User Story 2 - Googleカレンダーからの出社判定とシミュレーション (Priority: P1)

ユーザーは、システムが自身のスケジュール（出社か在宅か）をGoogleカレンダーから自動的に判定し、翌日の消費電力を正確に予測することを期待する。

**Why this priority**: 最適なSOCを算出するための前提となる、消費電力量の高精度な予測に直結するため。

**Independent Test**: Googleカレンダーにダミーの予定を登録し、システムが正しく「外出モード」または「在宅モード」を判定できたかを確認できる。

**Acceptance Scenarios**:

1. **Given** 翌日のGoogleカレンダーの予定に「【出社】」という文字列が含まれている時、 **When** システムが予定を取得する、 **Then** 翌日の状態を「外出モード」として認識し、それに応じた消費電力シミュレーションを実行する。
2. **Given** 翌日のGoogleカレンダーの予定に「【出社】」という文字列が含まれていない時、 **When** システムが予定を取得する、 **Then** 翌日の状態を「在宅モード」として認識し、それに応じた消費電力シミュレーションを実行する。

### User Story 3 - 1週間先のスケジュール・予測SOCビューアー (Priority: P2)

ユーザーは、アプリのメイン画面から向こう1週間の予定（出社/在宅）と、システムが計算した翌日以降の予測SOCの推移を一覧で確認したい。

**Why this priority**: ユーザーに安心感を与え、システム状態の可視化につながるが、自動化のコアロジックには影響しないためP2。

**Independent Test**: UI画面を開き、1週間分のリストがカレンダーおよび気象予測データに基づいて計算・表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** アプリのメイン画面を開いた時、 **When** 1週間先のスケジュールビューアーをスクロール表示する、 **Then** 日付ごとに「出社/在宅」の判定結果、天気予報（天気、雲量、降水確率）、および計算された予測SOCリストが正しく表示される。
2. **Given** スケジュールビューアーを表示した時、 **When** 日付を確認する、 **Then** 日付と曜日が「MM/dd (E)」の形式で表示される。
3. **Given** スケジュールビューアーを表示した時、 **When** 天気予報を確認する、 **Then** 天気の説明が日本語で表示され、雲量と降水確率が適切な改行位置で表示される。

### Edge Cases

- 端末がパスコードや生体認証で強力にロックされており、OSの制限により深夜の自動スクリーン点灯・ロック解除がブロックされた場合はどうなるか？
- 実行時刻周辺でネットワークの接続がなく、気象予報やGoogleカレンダーの取得に失敗した場合はどうなるか？（フォールバックとしてSOC 60%を設定する）
- マクロ実行中の20秒待機後の値再確認で、予期せぬエラーやUIの遷移遅延が発生しSOC設定が判定失敗した場合はどうなるか？

## Requirements

### Functional Requirements

- **FR-001**: システムは毎日指定された時刻に、ユーザーの介入なしにバックグラウンドで開始される自動タスクをスケジュールできなければならない。
- **FR-002**: システムは端末の画面オフやDoze（スリープ）状態から復帰して、マクロをフォアグラウンドで確実に実行できなければならない。
- **FR-003**: システムは連携されたGoogleカレンダーから対象日の予定タイトルや詳細を取得できなければならない。
- **FR-004**: システムは取得した予定から特定のキーワード（「【出社】」）の有無を判定し、「外出モード」と「在宅モード」に分類できなければならない。
- **FR-005**: システムは計算された結果を蓄電池へ反映するため、20秒待機後の値再確認によって成功・失敗を判定できなければならない。
- **FR-006**: システムはメイン画面にて、今日から起算して最大7日先までのスケジュール（出社/在宅判定）、天気予報、予測SOCをスクロール可能なリストで表示できなければならない。
- **FR-007**: 予測SOCの計算結果は、EP CUBEの仕様に基づき最低60%を下回らないように制限されなければならない。

### Key Entities

- **DailySchedule**: 日付、出社/在宅の判定結果（外出モード/在宅モード）、天気予報情報、予測SOCを保持するデータ構造。
- **AutomationExecutionResult**: 自動実行の成否、エラー詳細、実行時刻などを記録するログ情報。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 対象期間の連続した7日間において、ユーザーの手動操作が一切不要で深夜のマクロ設定処理が100%成功（20秒後の再確認クリア）すること。
- **SC-002**: Googleカレンダーの予定変更（【出社】の追加・削除）が、メイン画面のビューアーに10秒以内に反映され、予測リストが正しく表示されること。
- **SC-003**: 自動実行時、スリープ状態やロック状態に阻害されることなく、タスクの起動からマクロ実行完了までの処理が設定時刻から5分以内に完了すること。

## Assumptions

- 端末は設定画面等で必要な権限（ユーザー補助、他のアプリの上に重ねて表示、WakeLock等）が事前に全て許可されている。
- Googleカレンダーへのアクセス認証が事前に完了し、予定情報が読み取れる状態にある。
- ユーザーの予定のうち「【出社】」の文字列はタイトルや詳細の単純な文字列検索で一致判定可能である。
- ベースとなるマクロ操作（AccessibilityService）はすでに実機で動作確認済みである。
